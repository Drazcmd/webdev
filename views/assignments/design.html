In this assignment you will develop the frontend and functional design of your web application. (In the next assignment you will implement the full functionality of the frontend.)

Start by visiting your favorite social networking web application using Chrome Incognito (so you will not be automatically logged in). Take note of the landing page. Think about the positioning of the login and registration buttons. Can you register from the landing page or is there link to a separate page or modal popup? Similarly, can you log in directly from the landing page? Is the landing page devoid of content or is there a sampling of content? Using Chrome Incognito, compare the landing pages of Facebook and Twitter.

Next log into your favorite social networking web application and take note of the main page once logged in. Where are the user controls for logging out or changing your profile information. What content is display on the main page and how is it positioned relative to other components? How do you navigate around?

Think about these as you read the requirements below. Note that there is nothing to "submit" for the above two paragraphs.

Landing Page

In a previous assignment we constructed a simple landing page that had some of the key features for our social networking web application. But it is lacking style. Consider the layout of elements and what you will include on your landing page. Instead of modifying your simple page, create a new landing page, call it landing.html for consistency. Using raw CSS or a CSS Framework such as Bootstrap, Foundation, or SemanticUI, beautify your landing page. See below for the required components, but you are free to add more.

Main Page

Once a user logs in, their view will change. The logged in user should see certainly an option to log out, perhaps accompained by a means to change their profile (which contains a profile picture of course) and password. The main section of a user's page should be filled with posts from their feed. The feed is a combination of their own posts interspersed with posts from other users that are being followed. Prominently there is an area for the user to make a new post.

Also include a status headline for the user, and a sidebar with the followed users. Ideally we also see the status headlines of those users as well, although this may be delegated to context control (e.g., the headline appears when a user hovers their mouse over the followed user's name, ala Facebook). We need to be sure to separate the status headline from the new post card so that users are not confused by the two. The design layout and choices are up to you.

We also need a way to edit the followed list. Perhaps this is a button on the sidebar, or in a menu with the profile edit link. What about searching for posts? A search bar could be used to then replace the main section with the search results. But then we need to be sure to have a mechanism to get back to the default main page.

Create the main.html page that will act as the home page for the user once logged in. Using similar styling as the landing page, beautify your main page.

Functional Design

The frontend of our web application will communicate with the backend using a small collection of endpoints which constitute the API. Determine what that API will be, and which elements in your web application are the source for, and which are populated by the results from, each of the endpoints.

For example, the log in button will send the username and password to the server. The endpoint should be called /login and it should be a POST action.

Create a page design.html that contains a table of all of the endpoints, the verb associated with them (likely we will use GET, POST, PUT, and DELETE), the request payload that is sent, the reponse data that is returned, and a description(s) of the element(s) in your web application that are connected to them.

Your three pages should exhibit responsive layout (portrait vs landscape, desktop vs mobile) and have a unified style, i.e., look and feel. You should populate your pages with sample data, ala Lorem ipsum text and images, etc. Modals, if present, should open and close appropriately. Your site should exhibit a fluid user experience and be a site that you want to use.

You should write unobstrusive JavaScript and CSS. Have separate JS and CSS files for each HTML page as necessary -- do not combine disparate logic or styling into one file. However, there may be common functionality and styling that you share between the pages. In this case, have a common file in addition to the separate ones. It may be that you have no distinct styling, or perhaps only one or two CSS selector prescriptions, at which point it would be okay to only have the common CSS file and not separate ones for each page.
